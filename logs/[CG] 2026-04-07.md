# Meeting Notes — Journal Entry Rendering System

## Summary
We defined the core behaviour of the journal rendering system, focusing on preserving user meaning while improving readability. The key outcome is a clear separation between preprocessing (input-specific) and rendering (universal), anchored by a strict fidelity principle. We also refined terminology, clarified system boundaries, and identified where flexibility is appropriate vs risky.

---

## 1. Core System Model

The system operates in three layers:

1. **Preprocessing (input-specific)**
   - Audio → transcription  
   - Handwritten → OCR (text extraction from images)  
   - Typed → minimal/no preprocessing  

2. **Rendering (shared rules)**
   - Applies the same transformation rules to all inputs  
   - Focused on clarity without changing meaning  

3. **Optional Summarisation (separate layer)**
   - Used for weekly/monthly overviews  
   - Explicitly अलग from rendering to avoid meaning drift  

**Key decision:**  
Rendering rules must be **consistent across all input types** to maintain trust and output quality.

---

## 2. Core Principle: Fidelity

Rendering is governed by a single constraint:

> The system must preserve the original meaning of the source text.

Refinements:
- “Source text” replaces “transcript” to cover all input types  
- Fidelity means:
  - No new ideas
  - No interpretation
  - No emotional inference
  - No change in intent  

We moved away from vague language like “clarity” and replaced it with:

> Intervention may vary based on the presence of noise or errors (e.g. disfluencies, transcription errors, OCR mistakes), but must not alter meaning.

**Key nuance:**  
Flexibility is allowed at the *form level*, never at the *meaning level*.

---

## 3. What Counts as “Intervention”

Allowed:
- Remove filler words  
- Fix grammar and punctuation  
- Merge/split sentences  
- Light reordering (only if meaning is unchanged)

Not allowed:
- Adding new content  
- Reframing or interpreting  
- Inferring unstated emotions  
- Summarising within the rendered entry  

**Open tension discussed:**  
- Sentence restructuring improves readability  
- But increases risk of subtle meaning drift  
→ Needs careful constraint (light, not semantic)

---

## 4. Acceptable vs Not Acceptable Outputs

We established the need for **clear examples** to anchor behaviour.

Key distinction:
- Acceptable = same meaning, cleaner form  
- Not acceptable = compressed, interpreted, or reframed  

This becomes the system’s **implicit standard of comparison**.

---

## 5. Input Handling Philosophy

All inputs converge into **source text**, but:

- Audio & handwriting → higher noise → more correction needed  
- Typed → lower noise → more conservative handling  

**Important nuance:**
The system adapts **how much it edits**, not **what it is allowed to change**.

---

## 6. Transcript / Source Visibility

Decision:
- The rendered journal entry is the default view  
- The original input is accessible but not primary  

Behaviour:
- “View Original” toggle  
- User can edit original input  
- User can regenerate output  

**Implication:**  
The system is *editable and reversible*, not a black box.

---

## 7. Terminology Refinements

We corrected several terms to better match intent:

- “Transformation” → rejected (implies meaning change)  
- “Rendering” → adopted (format conversion, neutral)  
- “Fidelity” → defined as the governing principle  
- “Transcript” → replaced with “source text”  

**Key insight:**  
Naming shapes behaviour. Loose terms introduce system risk.

---

## 8. Design Philosophy (Implicit but Critical)

A few underlying principles emerged:

- **Trust > cleverness**  
  Users must feel their words are preserved, not rewritten  

- **Consistency > optimisation per input type**  
  One standard avoids unpredictable behaviour  

- **Separation of concerns**  
  Rendering ≠ summarisation  

- **Conservative by default**  
  Especially for cleaner inputs (typed text)

---

## 9. Open Questions / Future Considerations

- How far can sentence reordering go before it risks meaning drift?  
- Should users be able to toggle “strict fidelity” vs “more polished”?  
- How should summarisation be clearly separated in UI/UX?  
- Do we want version history for regenerated entries?

---

## Key Takeaway

The system is not a writing assistant — it is a **meaning-preserving renderer**.

Everything else (summarising, interpreting, enhancing) must be explicitly separated to protect that core function.
